## Card format

Each P-XX card contains:
- **Problem**: short description
- **Documented techniques to address it**: with source if possible
- **MMD status**: ALREADY COVERED (by what) / PARTIAL / TO INTEGRATE / N/A
- **MMD relevance**: High / Medium / Low
- **Integration**: where in the architecture if to be integrated

## Bundle A — Security

### P-03: Slopsquatting / package hallucination
- **Problem**: 19.7% of packages suggested by LLMs do not exist (USENIX 2025), and 43% of hallucinations are repeatable — attackers squat these names on npm/PyPI.
- **Techniques**: internal registry allowlist; existence + age + downloads check before `npm install`; systematic SBOM (Software Bill of Materials).
- **MMD status**: TO INTEGRATE.
- **Relevance**: High (Kid profile = code generated without human review).
- **Integration**: mandatory "deps gate" Worker before any install + constitution safe-by-default layer B.

### P-06: Lethal trifecta (Simon Willison)
- **Problem**: combination "private data + untrusted content + external comms" = guaranteed exfiltration via prompt injection.
- **Techniques**: "Agents Rule of Two" (max 2 of the 3 properties simultaneously); network egress allowlist sandbox; capability tokens.
- **MMD status**: TO INTEGRATE (Workers go on the internet + read user code + write files).
- **Relevance**: High.
- **Integration**: root-layer security constitution + Worker sandbox (network cut off except allowlist).
- **Source**: https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/

### P-07: Prompt injection via dependencies
- **Problem**: a library README can contain hidden instructions that the agent follows when reading the docs.
- **Techniques**: sanitization of third-party content; `untrusted` tagging in the prompt; two-LLM pattern (the planner does not see the raw content, only a safe representation).
- **MMD status**: TO INTEGRATE.
- **Relevance**: Medium-High.
- **Integration**: Watch Worker + Documentalist (provenance tag on all third-party content).

### P-08: Secrets leak
- **Problem**: agents commit `.env`, post API keys in logs/issues.
- **Techniques**: `git-secrets` / `trufflehog` as pre-commit hooks; log scrubbing; vault-only access (never a secret in cleartext in a prompt).
- **MMD status**: TO INTEGRATE.
- **Relevance**: High.
- **Integration**: constitution safe-by-default layer B + mandatory pre-commit hook.

## Bundle B — Correctness & sealed tests

### P-04: False positives "compiles but does not do the right thing"
- **Problem**: Karpathy (2026) — *"unvalidated execution"*: the test passes because the agent rewrote the test, not the code. SWE-bench Pro shows 24% of "verified" patches are actually incorrect (UTBoost study).
- **Techniques**: guarded (write-protected) tests; property-based testing; mutation testing; oracle independent from the coding agent.
- **MMD status**: PARTIAL (browser Reality Check exists, but no separate oracle).
- **Relevance**: High.
- **Integration**: "tester" Worker distinct from "coder" Worker + sealed tests at the start of the slice (BMAD Spec Phase).
- **Source**: https://arxiv.org/pdf/2509.16941

### P-05: Orthogonal damage
- **Problem**: Karpathy — the agent fixes X and breaks Y without knowing it. Measured: AI creates 1.7× more bugs than a human, +23.5% incidents post-adoption (Cortex study).
- **Techniques**: snapshot tests + full regression suite on every commit; blast radius computation (mapping the impact of a change before applying it).
- **MMD status**: TO INTEGRATE.
- **Relevance**: High (especially in brownfield mode).
- **Integration**: BMAD Orchestrator Validation phase + git diff guards.

### P-09: Behavioral correctness vs syntactic
- **Problem**: tests generated by the agent often just prove that the code does what it does, not what was asked.
- **Techniques**: spec-as-test (tests derived from `vision.md`/`slice.md` independently); LLM-as-judge on acceptance criteria; golden traces.
- **MMD status**: PARTIAL (spec injected but tests not systematically derived).
- **Relevance**: High.
- **Integration**: BMAD Spec Phase emits a sealed `acceptance.feature` before the Implement Phase.

## Bundle C — Observability + HITL tiered

### P-10: Eval harness for MMD itself
- **Problem**: how do we know if an improvement to the system is a regression? No internal benchmark.
- **Techniques**: reproducible "dream tests" suite (10 typical kid prompts + 10 pro); scoring time-to-MVP + cost + Reality-Check pass rate.
- **MMD status**: TO INTEGRATE (CRITICAL for self-improvement §7 of the scoping doc).
- **Relevance**: High.
- **Integration**: Observability module + nightly CI `dream-bench`.

### P-17: Calibrated approval gates
- **Problem**: HITL reduces critical errors from 23% to 5.1% (-78%) — but too many gates kills autonomy. Confidence-based routing is necessary.
- **Techniques**: tiered HITL (autonomous / notify / approve / co-edit); declarative + empirical confidence scoring.
- **MMD status**: PARTIAL (3 engagement modes exist but no per-action confidence calibration).
- **Relevance**: High.
- **Integration**: Conductor — each action carries a risk-score, routes to the appropriate engagement mode.

### P-18: Trust calibration (knowing when to challenge the AI)
- **Problem**: users (especially Kids) accept everything. No "I'm not sure" signal surfaced to the user.
- **Techniques**: self-uncertainty estimation; "second opinion" agent; visual surfacing of doubts ("⚠️ I'm not sure about this choice, validate?").
- **MMD status**: TO INTEGRATE.
- **Relevance**: High (Kid profile especially).
- **Integration**: Worker output schema = `{result, confidence, alternatives}`; Dream Catcher surfaces `confidence < 0.7` as a visual checkpoint.

### P-20: Observability of the agentic system
- **Problem**: without tracing, impossible to debug why a dream failed.
- **Techniques**: OpenTelemetry GenAI semconv; Langfuse / LangSmith; nested spans per sub-agent; LLM-as-judge scoring.
- **MMD status**: TO INTEGRATE.
- **Relevance**: High.
- **Integration**: cross-cutting Observability module (Conductor → Orchestrator → Worker traces).

## Bundle D — Living Specs + advanced context engineering

### P-01: Context rot / context pollution
- **Problem**: beyond saturation, the accumulation of noise (failed tool calls, verbose outputs, false starts) degrades quality before even reaching the limit. Anthropic talks about *"context rot"*.
- **Techniques**: structured compaction with a schema (vs free-form summary); structured note-taking in an external file; sub-agent dispatch to isolate dirty contexts.
- **MMD status**: PARTIAL (handoff at 70% exists, but no proactive intra-session cleanup).
- **Relevance**: High (long Ralph Loop sessions = strong drift).
- **Integration**: Conductor (triggers compaction at 50% and 75%) + Documentalist (structured note schema).
- **Source**: https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

### P-02: Constraint decay
- **Problem**: models excel at free generation, but their performance drops when explicit architectural constraints are imposed (arXiv 2025). "Constitution" rules are forgotten over the turns.
- **Techniques**: cyclic re-injection of critical constraints at the end of the prompt (recency bias); post-generation validation by deterministic rules (constitutional linter).
- **MMD status**: PARTIAL (constitution injected but not re-reminded during the Ralph loop).
- **Relevance**: High.
- **Integration**: Worker prompt template (constitution reminded at the start AND end) + constitutional linter as a post-step.

### P-21: Spec drift
- **Problem**: code evolves, `vision.md` becomes misleading → future sessions start from false premises. The entire MMD edifice rests on the reliability of these specs.
- **Techniques**: *living specs* (Augment Code) — the implementation rewrites the spec, diff highlighted for user validation.
- **MMD status**: TO INTEGRATE (CRITICAL).
- **Relevance**: High (auto-handoff and resumption rely on accurate specs).
- **Integration**: Documentalist — mandatory "spec reconciliation" phase at the end of the slice.

### P-22: Versioning & rollback
- **Problem**: massive bad generation → costly manual rollback.
- **Techniques**: commit-per-step (Aider pattern); ephemeral branches per slice; `undo dream` command.
- **MMD status**: PARTIAL (git used but no systematic commit-per-step).
- **Relevance**: Medium-High.
- **Integration**: granular Worker commits + Conductor exposes `rollback-slice`.

## Out of bundles — to consider but not v1 priority

### P-11: Cost runaway on long loops
- **Problem**: quadratic cost when naive (full re-serialization). Multi-agent = 4-15× tokens.
- **Techniques**: prompt caching (90% off on Anthropic); model routing (60-80% requests to cheap model); prompt compression (LLMLingua).
- **MMD status**: TO INTEGRATE.
- **Relevance**: High (Kid profile = zero budget).
- **Integration**: Conductor (router) + hard budget cap per dream.

### P-12: Model selection / fallback
- **Problem**: no single optimal model. Claude for architecture, GPT for algorithms, local Qwen-coder for mechanical refactor.
- **Techniques**: capability routing; fallback chain if rate-limited; continuous A/B.
- **MMD status**: TO INTEGRATE.
- **Relevance**: Medium-High.
- **Integration**: Conductor + routing table per task type.

### P-13: Semantic RAG vs grep (architecture decision)
- **Problem**: Cursor / Claude Code / Devin have dropped vector RAG — they use grep + AST + import chains. Vector RAG flattens the structure of the code.
- **Techniques**: AST chunking (cAST); graph-based retrieval (RANGER); agentic exploration (file tree → grep → read).
- **MMD status**: ARCHITECTURE DECISION TO LOCK IN.
- **Relevance**: High for brownfield.
- **Integration**: dedicated "explorer" Worker for brownfield mode, based on grep+AST+follow-imports (no vector RAG).

### P-14: Onboarding on an existing repo
- **Problem**: the agent rushes in without understanding the project's conventions.
- **Techniques**: mandatory "discovery" phase (reading README, CONTRIBUTING, package.json, sample tests); auto-generation of a project `CLAUDE.md`.
- **MMD status**: PARTIAL (brownfield constitution but no formal discovery phase).
- **Relevance**: High.
- **Integration**: BMAD Phase 0 for brownfield, output = `project-map.md`.

### P-15: Parallel conflicts (file-level)
- **Problem**: two workers edit the same file → broken merge.
- **Techniques**: git worktrees per agent (Cursor 2.0); CRDT for code; optimistic locking; Raft leader-election.
- **MMD status**: TO INTEGRATE if Conductor parallelism is enabled.
- **Relevance**: Medium (High if true parallelism).
- **Integration**: Orchestrator allocates one worktree per concurrent Worker.

### P-16: Semantic conflicts (vs file)
- **Problem**: two agents touch different files but the result is inconsistent (API contract changed on one side).
- **Techniques**: contract-first (OpenAPI / schema locked in spec phase); cross-cutting reviewer agent at the end of the cycle.
- **MMD status**: TO INTEGRATE.
- **Relevance**: Medium-High.
- **Integration**: Orchestrator merge phase + contract registry.

### P-19: Recovery on failure
- **Problem**: agent crash → state lost, run restarts from scratch.
- **Techniques**: systematic checkpointing (LangGraph); event-sourcing of the session; replay-able trace.
- **MMD status**: PARTIAL (state externalized but not event-sourced).
- **Relevance**: High.
- **Integration**: Conductor — store event log + `resume` command.

### P-23: Accessibility by default
- **Problem**: LLMs trained on broken web reproduce a11y defects (missing alt, missing ARIA). EU AI Act August 2026.
- **Techniques**: `A11Y.md` context file; WCAG 2.2 AA reviewer agents (CodeA11y); `axe-core` tests in Reality Check.
- **MMD status**: TO INTEGRATE.
- **Relevance**: High ("any human" mission = inclusive).
- **Integration**: safe-by-default constitution + Reality Check adds `axe-core`.

### P-24: i18n from day 1
- **Problem**: hardcoded English strings → costly overhaul later.
- **Techniques**: i18n-by-default lib + string linter.
- **MMD status**: TO INTEGRATE.
- **Relevance**: Medium.
- **Integration**: Pro profile constitution (optional for Kid).

### P-25: Generated performance / scalability
- **Problem**: code "works in dev" but N+1 queries, 5MB bundle, scroll jank.
- **Techniques**: perf budgets in the spec; Lighthouse in Reality Check; query analyzer.
- **MMD status**: PARTIAL.
- **Relevance**: Medium.
- **Integration**: extended Reality Check with perf scoring.

### P-26: Reproducibility / determinism
- **Problem**: same prompt = different code → impossible to debug, vague post-mortems.
- **Techniques**: seed + temperature 0 for critical steps; snapshot of resolved prompts; context hashing.
- **MMD status**: TO INTEGRATE.
- **Relevance**: Medium.
- **Integration**: Documentalist logs `prompt-hash + model-version + seed`.
