# ADR-004: FAST engine is a trimmed auto-dev, NOT a Ralph Loop

**Date**: 2026-05-17
**Status**: Accepted
**Authors**: Sébastien (project owner), auto-dev (Standard engine, v0.2 slice)

## Context

[MAKE_MY_DREAMS.md §3.1](../../MAKE_MY_DREAMS.md) introduced the **three engines, one brain** principle: FAST / STANDARD / DEEP, all sitting behind the same `mmd <dream>` surface, with a Mode Router (v0.2d) choosing between them based on context.

The question this ADR answers — settled in scoping v15 — is **what FAST actually IS under the hood**:

- Option A: a **Ralph Loop** — Geoffrey Huntley's minimalist `while :; do prompt | agent; done` pattern, possibly paired with the Anthropic `/ralph-loop` plugin.
- Option B: a **trimmed `auto-dev`** — the existing 4-phase pipeline, run with reduced ceremony (1× Party Mode instead of 3×, Phase 2 adversarial spec review skipped opportunistically, 1-page upfront spec).

v0.2 ships **Option B**.

## Decision

**FAST = trimmed `auto-dev` with a 1-page upfront spec.** Concretely:

- `mmd --fast "<dream>"` writes a deterministic 1-page spec (`lib/spec-derive.js`, ≤ 50 lines / ≤ 3000 chars) to `.mmd/shared/slice.md` BEFORE invoking auto-dev.
- The auto-dev subprocess is the same `claude -p /bmad-adv-auto-dev …` invocation as STANDARD, with two env-var-driven differences:
  - `MMD_AUTODEV_QUICK=1` in the subprocess env (passed through `lib/invoke-autodev.js#buildSubprocessEnv` via the `MMD_*` allowlist).
  - A `Engine: FAST (trimmed auto-dev …)` block appended to the prompt body that explicitly instructs the workflow to honor the quick-mode rules.
- The auto-dev workflow template (generated by `install-mmd.sh`) carries a `QUICK MODE` section that drops Phase 1 from 3× Party Mode to 1× and adds an opportunistic Phase 2 skip when the upfront spec passes a robustness heuristic (< 200 lines AND < 5 TODO/TBD/FIXME markers).
- Phase 3 (Implementation + 3-reviewer review) and Phase 4 (Final adversarial code review) are **unchanged**: correctness is non-negotiable per [`testing.md`](../../.specify/memory/constitution/testing.md) §III, and Phase 4 is cheaper to keep than to retroactively audit.

Ralph Loop is **NOT** shipped in v0.2. It remains a **deferred option for v0.6+**, gated behind a different flag if/when the empirical case for it is made (cf §"Future revisitation" below).

## Consequences

### Why trimmed auto-dev (Option B) over Ralph Loop (Option A)

**1. Empirical evidence from v0.1 + v0.2.5.**

The v0.1 walking skeleton ran the FULL auto-dev pipeline (4 phases, 3× Party Mode) and delivered a working PWA — drawing app overlaying an image on the camera feed — in ~45 minutes. v0.2.5 then ran auto-dev twice on the `mmd serve` slice. Both runs produced quality output with adversarial reviews flagging real issues that improved the code. Sébastien's direct experience: **auto-dev converges; Ralph Loop's convergence properties are weaker in practice.** Geoffrey Huntley's writeup describes Ralph as effective when the goal is "tweak one thing on existing code"; trimmed auto-dev is effective across a wider range of dreams, including small new features.

**2. Investment reuse.**

Auto-dev exists. It's tested. It's integrated. It honors the constitution. It produces dream-bench-grade output. A FAST engine that reuses 90% of auto-dev's infrastructure (constitution injection, fresh-context sub-agents, 3-reviewer adversarial loop in Phase 3) ships in v0.2. A FAST engine that wires up Ralph from scratch would consume v0.2's entire budget and arrive with no track record.

**3. Convergence safety.**

A pure Ralph Loop with no built-in checkpoints can spin forever or drift away from the goal — both observed by Huntley and acknowledged in scoping v15 §3.1. Auto-dev's Phase 3 (3-reviewer adversarial review with loopback) and Phase 4 (final adversarial code review) are explicit convergence guards: any loopback that doesn't converge after 5 iterations escalates to the user. That safety mechanism is gone in pure Ralph.

**4. Audit and dream-bench compatibility.**

`status.json.engine_metrics` was designed assuming auto-dev's 4-phase structure (`party_mode_rounds`, `phase2_skipped`, …). A Ralph Loop has no phases — making before/after FAST vs STANDARD comparisons (the seed of dream-bench in v0.2b) much harder to define. Keeping FAST as a trimmed variant of STANDARD makes the metrics directly comparable.

**5. Consistency with ADR-003's "pragmatic vanilla" precedent.**

[ADR-003](./003-mmd-serve-vanilla-stack.md) chose vanilla HTML/CSS/JS + SSE over a framework + WebSocket because the existing minimal stack was sufficient. Same reasoning applies here: the existing auto-dev pipeline, trimmed, is sufficient for the FAST use case. Adding a second orchestration paradigm (Ralph) is only justified if the first one demonstrably fails — which it has not.

### Trade-offs accepted

- **Ceremony floor**: even trimmed, FAST runs Phase 1 + (optionally) Phase 2 + Phase 3 (with 3-reviewer review) + Phase 4. The ceremony floor is higher than pure Ralph. For a "tweak this one thing in 30 seconds" iteration this is overkill — that's the niche Ralph would fill in v0.6+ if data supports it.
- **Spec-derivation heuristic is conservative**: `lib/spec-derive.js` uses keyword matching (button, form, camera, draw, upload, color, save) to infer 1–3 ACs. A dream that uses none of those words falls back to a generic "feature is functional + visible + no regression" AC. False negatives (matching too few keywords) tend to produce overly-conservative specs; false positives produce ACs that read weirdly. Both are visible to the user in `slice.md` and easy to refine manually with `--standard` if needed.
- **No `--quick-mode` flag at the auto-dev CLI level**: we pass the directive via env var + prompt body rather than a real CLI flag, because the `bmad-adv-auto-dev` skill doesn't expose flags — it's invoked via `claude -p`. The two-channel approach (env var + prompt) is belt-and-suspenders against the LLM ignoring one or the other.

### Future revisitation (v0.6+ candidate trigger)

Open a follow-on ADR proposing Ralph Loop as a 4th engine (or as a sub-mode of FAST) if any of the following is observed:

- The FAST engine fails to converge on a category of dreams that intuitively fit Ralph's "tweak one thing" niche.
- dream-bench (v0.2b+) shows FAST's `duration_seconds` consistently exceeds 15 min for trivial brownfield dreams — at which point trimming further inside auto-dev would mean cutting Phase 3/4 (unacceptable) so we'd need a separate engine.
- The Anthropic `/ralph-loop` plugin matures into a turnkey integration with built-in convergence guards.

Until then: FAST = trimmed auto-dev, full stop.
